================================================================================
Aetheria Companion Core Engine & Frontend Sync Architecture Summary
================================================================================

This document outlines the detailed logic steps, objectives, and implementation
details of each modified/created file in the project. It describes how they
connect together and specifies all external APIs integrated into the system.

================================================================================
1. EXTERNAL APIS USED
================================================================================
* Gemini Generative Language API
  - Endpoint: https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent
  - Objective: Advanced intent classification, entity extraction, and structured JSON generation.
  
* Groq Chat Completions API
  - Endpoint: https://api.groq.com/openai/v1/chat/completions
  - Model: llama-3.1-8b-instant
  - Objective: Ultra-fast conversation generation and web search results summarization.
  
* Hugging Face Inference API
  - Primary Model: sentence-transformers/all-MiniLM-L6-v2 (384 dimensions)
  - Fallback Model: BAAI/bge-small-en-v1.5 (384 dimensions)
  - Objective: Text feature extraction for RAG cognitive memory vectors.
  
* DuckDuckGo Search API
  - Package: duckduckgo_search (resolves to the 'ddgs' package)
  - Objective: Free, credential-less search results mapping title, snippet, and link.

* n8n / Make Webhooks Integration
  - Objective: Asynchronous dispatch of automation payloads (emails, reminders, resume JSON metadata).

3. DETAILED FILE IMPLEMENTATION DETAILS
================================================================================

--------------------------------------------------------------------------------
File: src/context/ChatContext.tsx [Frontend Context]
--------------------------------------------------------------------------------
* Objective: 
  Manages client-side authentication, companions list, active chat threads,
  memories list, and reconnection/synchronization routines.
* Logic Steps:
  1. Reconnection Poller: If backendOffline is true, starts a 5-second interval timer
     and registers a window "online" listener to ping the backend root.
  2. Reconcile Sync: Once connection returns, sends the localStorage queue to
     POST /api/v1/chat/sync, clears the local storage on success, and re-syncs all
     active data from the DB.
  3. changeThreadMode: Takes threadId and sessionMode, makes a PUT request to
     `/chat/threads/{threadId}` with the updated session mode, updating database records.

--------------------------------------------------------------------------------
File: src/components/Sidebar.tsx [Dashboard Navigation Sidebar]
--------------------------------------------------------------------------------
* Objective:
  Provides navigation links, list of active chat threads, and a visual Workspace Mode selector.
* Logic Steps:
  1. Mount Synchronization: Reads the fallback mode from localStorage key
     `aetheria_default_session_mode` on mount.
  2. Active Mode Binding: Sets active mode selection to `activeThread.sessionMode` (if it
     exists), otherwise falls back to `localDefaultMode`.
  3. Mode Selection Change: Clicking a workspace mode (Personal, Professional, Academic,
     Researcher, Playground) writes the selection to localStorage and, if a thread is active,
     calls the `changeThreadMode` API to sync changes with the database.
  4. Parameter Forwarding: Creating a new thread via `handleCreateNewChat` automatically
     forwards the active workspace mode as the third parameter to `createNewThread()`.
  5. Active Tags: Renders a sub-label (e.g. "PERSONAL MODE") under each thread title.

--------------------------------------------------------------------------------
File: src/components/ChatBubble.tsx [Message bubble renderer]
--------------------------------------------------------------------------------
* Objective:
  Displays user and AI messages and renders rich elements like code boxes and action buttons.
* Logic Steps:
  1. URL Scrubbing: Uses a regex check matching `(?:pdf_url|download_url|resume_url)\s*:\s*(https?://[^\s]+)`.
     If a match is found, it extracts the URL, removes the matched text line from the visible message
     to avoid showing raw links, and stores the URL in a React variable.
  2. Glassmorphic Rendering: If the URL is found, appends a responsive bouncing download button
     labeled "Download Your Resume", linking to the dynamic PDF document.

--------------------------------------------------------------------------------
File: backend/app/services/pdf_generator.py [Dynamic PDF Compiler]
--------------------------------------------------------------------------------
* Objective:
  Generates a high-quality, professionally formatted resume PDF on the local server using ReportLab.
* Logic Steps:
  1. Setup Document: Uses a SimpleDocTemplate with 0.5-inch margins for compact spatial formatting.
  2. Branding: Applies a customized color palette using HexColors (Primary: Dark Slate #0F172A,
     Secondary: Muted Blue #475569, Accent: Cyan #06B6D4 matching Aetheria theme).
  3. Flowables Story:
     - Header: Centers the Candidate Name in Helvetica-Bold (24pt) and contact info separated by bullets.
     - Section Headings: Appends uppercase section titles followed by a subtle cyan horizontal divider line.
     - Bullet Parsing: Splits section content by newlines, checks for bullet qualifiers ('-' or '*'),
       and renders nested bullet paragraphs with proper indentation.
  4. Compilation: Invokes `doc.build(story)` to compile the flowables into a PDF file on disk.

--------------------------------------------------------------------------------
File: backend/app/api/workflow.py [Workflow API Router]
--------------------------------------------------------------------------------
* Objective:
  Exposes workflow execution and dynamic file serving endpoints.
* Logic Steps:
  1. Static Resumes Dir: Initializes `static_resumes` folder in the backend workspace.
  2. FileResponse Route: Exposes a GET route `/resume/download/{filename}`.
     It verifies that the filename exists in the static folder, reads it, and returns the binary
     stream as a FileResponse with `media_type="application/pdf"`.

--------------------------------------------------------------------------------
File: backend/app/services/agent_service.py [Agent Intent Router & Pipeline Manager]
--------------------------------------------------------------------------------
* Objective:
  Routes intent execution, performs validations, and executes professional workflows.
* Logic Steps:
  1. Fallback Intent Checks: If Gemini is offline, runs regex checks matching resume keywords or counting
     spaced label tags (e.g. `Name :`).
  2. Segment Offset Parser (Attempt 3): If user pastes a raw copy of their CV, segments sections dynamically
     by sorting header index coordinates, allowing extraction from newline-stripped formats.
  3. validation check: Verifies that Name, Contact, Education, Experience, and Skills are present. If
     any is missing, compiles already-known details and prompts the user with a pre-filled template.
  4. Local PDF compiler fallback: Once details are complete, triggers `generate_resume_pdf` to create the
     actual resume PDF file, naming it `resume_{uuid.uuid4().hex[:8]}.pdf`. Returns the local server URL
     `http://localhost:8000/api/v1/workflow/resume/download/{filename}` as the download link.
  5. Webhook dispatch: If `N8N_WEBHOOK_URL` is set, posts the JSON payload to n8n. If the webhook returns
     a URL, uses it; otherwise falls back to the local PDF.

--------------------------------------------------------------------------------
File: backend/app/services/emotion_service.py [Sentiment Analytics]
--------------------------------------------------------------------------------
* Objective:
  Tracks sentiment intensity and routes buddy mode hinges.
* Logic Steps:
  1. Hinglish/Slang Analysis: Analyzes the message text for Hindi/Hinglish slang or venting terms.
  2. Bhai Mode Adapter: If personal mode is active and slang is detected, instructs the LLM
     to drop formal preambles and talk like a close friend, inserting Hinglish slang terms naturally.

4. VERIFICATION SCRIPTS (Located in backend/scratch/)
================================================================================
* verify_pasted_resume.py: Tests raw copy-pasted CV text parsing and PDF generation.
* verify_resume_api.py: Verifies template label matching containing extra colon spaces.
* test_pdf_gen.py: Direct ReportLab compile verification.
* test_session_mode.py: Verifies database migrations and default personal mode parameters.
* test_dynamic_mode_routing.py: Tests instructions for Academic, Researcher, and Professional modes.
* check_render_threads.py: Diagnostics verifying Render backend database and API integrity.
